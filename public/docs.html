Free, privacy-first form backend you can self-host.

FormForge is a FormZero-style backend for static sites, upgraded for a 2026 Cloudflare Workers + Next.js/OpenNext + D1 workflow. It does not require DATABASE_URL, PostgreSQL, Neon, Supabase, or any paid subscription.

Features

  • Submit HTML forms with a simple action URL.
  • Submit JSON via fetch, XHR, Astro, React, Vue, Svelte, or plain JS.
  • Unlimited forms and submissions based on your own Cloudflare D1 limits.
  • D1-backed users, sessions, forms, submissions, API keys, notifications, and audit logs.
  • CSV export endpoint for every form.
  • Honeypot spam trap.
  • Origin allowlist for browser submissions.
  • Optional proof-of-work challenge support.
  • Optional webhook and email notifications.
  • Strict build safety: no DB initialization at module top level.

Architecture

Frontend: Next.js 16 App Router, React 19, Tailwind CSS. Runtime: Cloudflare Workers via @opennextjs/cloudflare. Database: Cloudflare D1 through drizzle-orm/d1. Schema: SQLite tables from drizzle-orm/sqlite-core.

Important: The database client is created only inside request handlers with getDb(). If Cloudflare bindings are not available during next build, FormForge returns safe diagnostics instead of throwing.

One-click deployment

When someone clicks the deploy button, Cloudflare asks that person to log in with their own Cloudflare account. The Worker, D1 database, secrets, and generated repository belong to them, not to the original GitHub username.

  1. Click Deploy to Cloudflare.
  2. Sign in to your free Cloudflare account. No credit card is required for the free tier.
  3. Choose a Worker/project name, for example formforge or my-forms.
  4. Cloudflare automatically creates the formforge-db D1 database. The database_id in wrangler.jsonc is intentionally blank so Cloudflare provisions it for you.
  5. Set AUTH_SECRET (generate with openssl rand -hex 32). Leave email variables blank if you do not need email.
  6. Deploy. On the first request FormForge auto-creates its tables, so no manual migration step is needed.
  7. Open /dashboard on your new Workers URL, register an owner account, and create your first form.

Note: The deploy button may copy the seed repository into the deployer's GitHub/GitLab account. This is expected; it is how the deployer owns their deployment.

Environment variables

NameRequired?What to enter
AUTH_SECRETYes for authGenerate a long random value: openssl rand -hex 32. In Cloudflare dashboard: Worker → Settings → Variables → Add secret.
RESEND_API_KEYNoOnly if you enable email notifications. If blank, email is skipped automatically.
RESEND_FROMNoOnly with RESEND_API_KEY. Example: FormForge <forms@example.com>.
DBYesThis is not a secret text variable. It is a Cloudflare D1 binding named DB configured in wrangler.jsonc or the Cloudflare dashboard.

Do not add DATABASE_URL. FormForge intentionally avoids PostgreSQL for the default Cloudflare deployment.

D1 database setup commands

For the one-click flow, Cloudflare provisions the database automatically. For manual or local control, run these after logging in with npx wrangler login:

# Optional: create D1 manually (the deploy button does this for you)
npx wrangler d1 create formforge-db
# copy the returned database_id into wrangler.jsonc if not using auto-provisioning

npx drizzle-kit generate
npx wrangler d1 migrations apply DB --remote
npm run deploy

If Cloudflare asks for a database name, use formforge-db. If your Worker name is different, that is okay; the D1 binding name should remain DB. FormForge also creates tables on first request, so migrations are a backup, not a requirement.

API and usage examples

1) HTML form

<form method="POST" action="https://YOUR-WORKER.workers.dev/api/submit/endpoint_xxx">
  <input name="email" type="email" required>
  <textarea name="message" required></textarea>
  <input name="website" hidden tabindex="-1" autocomplete="off">
  <button>Send</button>
</form>

2) JSON submission

await fetch("https://YOUR-WORKER.workers.dev/api/submit/endpoint_xxx", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email: "hello@example.com", message: "Hi!" })
});

3) Owner APIs

  • POST /api/auth/register — create owner account.
  • POST /api/auth/login — sign in.
  • GET /api/forms — list forms.
  • POST /api/forms — create a form.
  • GET /api/forms/:formId/submissions — list submissions.
  • GET /api/forms/:formId/export — CSV export.

Privacy and security

  • Data is stored in the deployer's D1 database.
  • No built-in third-party analytics or tracking pixels.
  • IP addresses are hashed per form when stored.
  • Session tokens and API keys are HMAC-hashed before storage.
  • Passwords use PBKDF2-SHA256 via Web Crypto.
  • Email/webhook integrations are optional and disabled by default.

Troubleshooting

Error: DATABASE_URL is required

This happens when old PostgreSQL code imports a DB pool at module top level. FormForge fixes it by removing pg, using drizzle-orm/d1, and calling getDb() inside each route handler.

D1 not configured

Make sure wrangler.jsonc has d1_databases with binding DB, database name formforge-db, and your real database_id.

Environment variable warnings in preview

AUTH_SECRET is needed for login/register. RESEND_API_KEY and RESEND_FROM are optional; leave blank to disable email. Never put production secrets into public previews.

Does it use my GitHub username?

No. The source repo URL is only a template. The deployer logs in to their own Cloudflare account and owns the Worker/D1 resources created during deployment.
